# W5-04 / Issue #47 Student Feedback And Notification Evidence

## Scope

Issue #47 adds real backend APIs for:

- student feedback (`student_feedback`)
- notifications (`notifications`)
- notification recipients (`notification_recipients`)

Branch: `feature/feedback-notifications`

## API Contract

Student feedback:

- `GET /api/student-feedback`
  - roles: SV, CBDT, QTV
  - filters: `status`, `feedback_type`, `lab_schedule_entry_id`
  - SV sees only their own feedback; CBDT/QTV can list submitted feedback from all students.
- `POST /api/student-feedback`
  - role: SV
  - creates a `submitted` feedback row tied to a published schedule entry visible to the logged-in student.
  - body: `lab_schedule_entry_id`, `feedback_type`, `content`, optional `contact_info`.
  - creates unread notifications for active CBDT/QTV users.
- `PATCH /api/student-feedback/:id`
  - roles: CBDT, QTV
  - updates `feedback_status`, handler metadata, and optional `response_text`.
  - `responded` and `closed` updates create an unread notification for the original student.

Notifications:

- `GET /api/notifications`
  - roles: any logged-in user
  - returns only notifications for the current user.
  - filters: `status`, `notification_type`.
- `PATCH /api/notifications/:id/read`
  - roles: any logged-in user
  - marks one current-user notification as `read`.
- `PATCH /api/notifications/:id/acknowledge`
  - roles: any logged-in user
  - marks one current-user notification as `acknowledged`.
- `PATCH /api/notifications/read-all`
  - roles: any logged-in user
  - marks all unread current-user notifications as `read`.

## Important Behavior

- Student feedback is accepted only for published schedule entries that belong to one of the student's practice teams.
- Notification ownership is enforced through `notification_recipients`; users cannot read or update another user's notification state.
- Feedback reviewer roles are limited to CBDT and QTV.
- The module uses the existing schema; no migration is required.

## Frontend Wiring Note

The existing mock pages can be wired without changing the main workflow shape:

- student schedule/detail UI:
  - submit feedback through `POST /api/student-feedback`
  - list own feedback through `GET /api/student-feedback`
- CBDT/QTV feedback management UI:
  - load submitted feedback through `GET /api/student-feedback?status=submitted`
  - answer feedback through `PATCH /api/student-feedback/:id`
- notification UI:
  - list notifications through `GET /api/notifications`
  - update item state through `PATCH /api/notifications/:id/read` and `PATCH /api/notifications/:id/acknowledge`
  - bulk read through `PATCH /api/notifications/read-all`

Frontend integration remains tracked by #49.

- Smoke-created feedback and related notification rows were cleaned up after the run.

## Limitations

- #47 only implements backend APIs and evidence. UI wiring remains tracked by #49.
- Notifications are created for feedback submission and reviewer response only. Other event types can reuse the same notification service pattern later.
- `PATCH /api/notifications/read-all` intentionally affects only unread notifications for the current user; acknowledged notifications remain acknowledged.
